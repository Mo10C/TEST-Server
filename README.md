# TEST-Server

このプロジェクト、実は「専用サーバー」を自前で持つ構成じゃなくて、静的ファイル配信 + Firebase(サーバーレス) という構成なので、ボトルネックになる場所がいくつかに分かれています。整理すると:
構成の内訳
コンポーネント役割実体index.html / app.js / styles.cssアプリ本体静的ファイル(どこかのホスティング)Firestore (sim_seed_stats, sim_users, sim_meta)シード統計・ユーザー登録・管理者情報Google FirebaseCloudflare Worker (riotProxyUrl)Riot API中継CloudflareDiscord OAuthアカウント連携Discord
つまり「このサーバー」の耐久性 = 一番弱いパーツの上限で決まります。
各パーツの上限
① 静的ファイル配信
GitHub Pages / Cloudflare Pages系なら実質無制限に近い(CDN配信)。ここはまず問題になりません。
② Firestore（一番のボトルネック）
SEED_STATS_CONFIGが無料プラン(Sparkプラン)のままだと:

書き込み: 1日20,000件まで
読み取り: 1日50,000件まで
保存容量: 1GB

submitSeedRecordはシミュレーション1回につき1書き込み、fetchSeedRecordsは結果閲覧のたびに読み取りが飛びます。つまり1日あたり同時多発で数千〜2万プレイ程度が上限の目安です。それを超えると新規の統計送信・閲覧が失敗し始めます（コード上はcatchで握りつぶしてローカル保存にフォールバックする作りなので、ユーザー体験としては「共有統計だけ表示されなくなる」形で緩やかに壊れます）。
③ Cloudflare Worker（Riot ID連携）
Workers無料枠は1日100,000リクエストですが、実質のボトルネックはRiot Games APIのレート制限（開発者用キーだと20req/秒・100req/2分など非常に厳しい）です。同時に大量のユーザーがRiot ID連携をすると、ここが先に詰まる可能性が高いです。
④ セキュリティ面の懸念（容量とは別問題ですが重要）
sim-config.jsのFirebase APIキーはクライアント側に露出していますが、これはFirebase設計上「公開されて問題ない」もの——本当の防御はFirestoreのセキュリティルール側です。コード内コメントに「該当コレクションに read/write を許可するルールが必要」とあるので、もしルールが緩い(誰でも書き込み可)設定になっていると、悪意のあるユーザーがAPIキーを使って大量の不正書き込みを送りつけ、無料枠を一瞬で食い潰す・あるいは有料プラン(Blaze)なら課金が発生するリスクがあります。
まとめると

通常利用の想定（友人内・小規模コミュニティ）ならSparkプラン無料枠で全く問題なし
数百人規模で毎日プレイされるような広がり方をすると、Firestoreの書き込み上限が先に頭打ちになりやすい
本格的に耐久性を上げたいなら Blazeプラン(従量課金)への切り替え＋Firestoreセキュリティルールの厳格化（例: レート制限、必須フィールド検証）が現実的な対策です

セキュリティルールの現状や、実際のFirebaseコンソールの使用量を確認したいなら、そのあたりを一緒に見ることもできますよ。
